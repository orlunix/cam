# Dev Log — 2026-03-31

## Current Working Model: cam ↔ camc Agent Lifecycle

### Architecture Overview

```
Mobile APP / Teams (TeaSpirit)
        ↓
   cam serve (API + Relay)
        ↓  reads
   SQLite cam.db ←── CamcPoller (every 5s)
                          ↓ polls by machine
        ┌─────────────────┼─────────────────┐
        ↓                 ↓                 ↓
   local camc        remote camc       remote camc
   (read JSON)       (SSH + camc)      (SSH + camc)
        ↓                 ↓                 ↓
   ~/.cam/agents.json  agents.json      agents.json
```

### Agent ID

8-char hex, globally unique. Generated by camc via:

```python
uuid5(NAMESPACE_DNS, hostname + timestamp + random).hex[:8]
```

- `hostname` prevents collisions across NFS-shared home directories
- `timestamp + random` prevents collisions on the same machine
- 8-char is the canonical ID everywhere: tmux session names (`cam-{id}`),
  display, prefix matching, cross-machine references

cam server also has legacy full-UUID agents (36-char) from before camc
delegation. `AgentStore.get()` supports prefix matching so 8-char IDs
resolve to full-UUID records via `WHERE id LIKE ?%`.

### CamcPoller: Machine-Based Polling

CamcPoller iterates **machines** (from `~/.cam/machines.json`), not contexts.
Each machine is polled once per cycle regardless of how many contexts it hosts.

```
machines.json → for each machine:
  local?  → read ~/.cam/agents.json directly
  remote? → SSH: camc --json list
```

Fallback: if `machines.json` doesn't exist, derives unique machines from
the context store (deduplicated by `user@host:port`).

For each agent returned by camc:
1. **`store.get(agent_id)`** — prefix match finds existing record
2. **If found** — update status/state if changed
3. **If not found** — check tmux_session collision (shadow agent), else import
4. **Import uses agent's own data** — `context_name`, `context_path`, etc.
   come from the agent record itself, not from the polling context/machine

### Agent Data Flow

```
camc run → creates tmux session → writes to agents.json (immediately)
         → spawns monitor subprocess
         → monitor updates status/state in agents.json

CamcPoller → reads agents.json (local) or SSH camc list (remote)
           → upserts into SQLite by agent ID
           → emits events to EventBus

cam list   → reads SQLite directly (no limit, shows all agents)
cam API    → reads SQLite (no default limit)
TeaSpirit  → calls `cam --json list` CLI → reads SQLite
```

### NFS Cluster Considerations

On NVIDIA PDX/DC7 containers, multiple machines share `~/.cam/agents.json`
via NFS. Each machine's camc filters by `hostname` field:

- `camc list` only shows agents from current hostname
- `camc heal` only touches agents from current hostname
- Agent ID includes hostname in uuid5 seed, preventing collisions

But when cam server reads `agents.json` directly (local poll), it sees
**all** agents from all NFS-sharing machines. The poller imports them
using each agent's own `context_name`, not the machine being polled.

## Changes Made Today

### 1. camc: Name/index matching for agent lookup

**Files**: `src/camc_pkg/storage.py`, `src/camc_pkg/cli.py`

`AgentStore.get()` matching order:
1. Exact ID match
2. Exact tmux_session match
3. ID prefix match
4. Name match (exact, then prefix, case-insensitive)
5. Numeric index (1-based, matches `camc list` order)

Added `a` as alias for `attach` (expanded in `sys.argv` before argparse).

### 2. camc: Save agent record before startup wait

**File**: `src/camc_pkg/cli.py`

Moved `store.save()` to right after tmux session creation, before the
30-second startup readiness loop. Agent now appears in `camc list`
immediately, not after startup completes.

### 3. camc: Reordered help examples

**File**: `src/camc_pkg/cli.py`

Top three examples are now the core workflow:
```
camc run -n myfix "fix lint errors"
camc list
camc a myfix
```

### 4. cam: Remove default limit on agent listing

**Files**: `src/cam/cli/agent_cmd.py`, `src/cam/api/routes/agents.py`

- `cam list` default limit: 50 → 0 (all)
- API `/api/agents` default limit: 20 → 0 (all)

Root cause: 59 agents in SQLite, limit=50 truncated 9 running agents.
TeaSpirit (which calls `cam --json list`) was showing incomplete results.

### 5. cam: CamcPoller polls by machine instead of context

**File**: `src/cam/core/camc_poller.py`

Before: iterated contexts, deduplicated by `user@host:port`, used context
name/ID when importing agents. Broke for NFS-shared agents whose
`context_name` didn't match the polling context.

After: reads `~/.cam/machines.json` directly, one poll per machine.
Agent import uses `_camc_agent_to_model(agent_data)` with no context
override — agent's own `context_name` is authoritative.

Changed poll result log from `logger.debug` to `logger.info` for
visibility in cam serve logs.
